# JMeter 压测使用指南 — Atlas 热点题库接口

## 一、前置条件

| 条件 | 检查方式 | 备注 |
|------|---------|------|
| JDK 17+ | `java -version` | 无 JDK 则 JMeter 无法启动 |
| JMeter 5.6+ | `jmeter -v` | [下载地址](https://jmeter.apache.org/download_jmeter.cgi)，解压即可，无需安装 |
| 应用已启动 | `curl http://localhost:8080/api/v1/question-banks/1/hot-practice-detail` | 应返回 200 + 15 道题 JSON |
| Redis 已运行 | `redis-cli ping` | 返回 PONG |
| 测试数据已导入 | `mysql -u root -p -e "SELECT COUNT(*) FROM quiz_atlas.question"` | > 0 即可 |

## 三、脚本结构速览

打开 `.jmx` 后看到的结构：

```
Atlas-热点题库压测-对照实验
├── User Defined Variables          ← 全局参数（端口、线程数、时长等）
└── 压测线程组 (100 并发, 300s)
    ├── HTTP 请求默认值              ← 统一配置域名/端口/编码
    ├── GET 热点题库详情              ← 被测请求
    │   ├── 断言-HTTP 200            ← 必须返回 HTTP 200
    │   ├── 断言-业务code=200        ← JSON 中 $.code == 200
    │   └── 断言-bankId 存在         ← JSON 中 $.data.bank.id 不为空
    ├── 察看结果树                   ← 调试用，CLI 压测可关闭
    ├── 聚合报告                     ← 压测结束查看 Summary
    └── Summary Report
```

### 全局变量说明

| 变量 | 默认值 | 作用 |
|------|--------|------|
| `TARGET_HOST` | `localhost` | 被测应用 IP/域名 |
| `TARGET_PORT` | `8080` | 被测应用端口 |
| `BANK_ID` | `1` | 压测的题库 ID |
| `THREADS` | `100` | 并发线程数 |
| `RAMP_UP` | `30` | 线程启动耗时（秒） |
| `DURATION` | `300` | 持续压测时长（秒） |
| `PREWARM_URL` | 完整 URL | 预热用 curl 地址（见实验甲） |
| `REDIS_DELETE_KEY` | `smart_quiz:bank_detail:1` | 冷缓存实验用的 Redis Key |

**命令行覆盖参数**（无需改 `.jmx`）：
```bash
jmeter -n -t jmeter/atlas-hot-bank-stress-test.jmx \
  -Jthreads=200 -Jramp_up=60 -Jduration=600 \
  -l result.jtl -e -o report/
```

## 四、对照实验：实验甲 vs 实验乙

两组实验**用完全相同的脚本、参数、环境**，唯一变量是「缓存是否命中」。

### 4.1 实验甲 — 缓存命中（体现优化后）

**步骤：**

```bash
# 1. 确保应用、MySQL、Redis 全部正常运行

# 2. 预热 Redis 缓存（连续请求 3 次，让热点 key 写入）
curl http://localhost:8080/api/v1/question-banks/1/hot-practice-detail
curl http://localhost:8080/api/v1/question-banks/1/hot-practice-detail
curl http://localhost:8080/api/v1/question-banks/1/hot-practice-detail

# 3. 验证缓存已写入
redis-cli EXISTS smart_quiz:bank_detail:1
# 应返回 1

# 4. 启动压测（CLI 模式，GUI 不开！）
jmeter -n -t jmeter/atlas-hot-bank-stress-test.jmx \
  -l result-cache-hit.jtl \
  -e -o report-cache-hit/

# 5. 报告在 report-cache-hit/index.html，浏览器打开查看
```

### 4.2 实验乙 — 冷缓存 / 强制回源（对照基线）

**步骤：**

```bash
# 1. 删除目标题库的缓存 Key（让下次请求走完整回源链路）
redis-cli DEL smart_quiz:bank_detail:1

# 2. 可选：也删除互斥锁 Key（如果你的代码中锁 Key 是 smart_quiz:bank_lock:1）
redis-cli DEL smart_quiz:bank_lock:1

# 3. 确认缓存已清空
redis-cli EXISTS smart_quiz:bank_detail:1
# 应返回 0

# 4. 启动压测（同参数，输出到不同文件）
jmeter -n -t jmeter/atlas-hot-bank-stress-test.jmx \
  -l result-cold-start.jtl \
  -e -o report-cold-start/

# 5. 报告在 report-cold-start/index.html
```

> **原理**：删 Key 后，首轮请求会穿透到 MySQL 并回写 Redis。
> 实验乙的「冷启动阶段」RT 会明显偏高（查库 + 锁竞争 + 写回缓存）。
> 如果你只想看完全无缓存的极端情况，可以在压测全程持续删 Key（见 4.3）。

### 4.3 持续冷缓存（可选，更极端）

如果你想让**整个压测期都走 MySQL**（模拟「Redis 挂了但服务不崩」的极限场景），可以持续删 Key：

```bash
# 在另一个终端跑这个脚本，每秒删一次缓存 Key
while true; do
  redis-cli DEL smart_quiz:bank_detail:1
  sleep 1
done

# 然后启动压测
jmeter -n -t jmeter/atlas-hot-bank-stress-test.jmx \
  -l result-no-cache.jtl \
  -e -o report-no-cache/
```

> **注意**：此时 MySQL 压力极大，先确认 MySQL 的 `max_connections` 足够（建议 ≥ 200），
> 否则可能出现连接池耗尽导致错误率飙升。

## 五、进阶玩法：阶梯找「并发拐点」

逐步加压，观察错误率何时突破 0%：

```bash
# 50 并发
jmeter -n -t jmeter/atlas-hot-bank-stress-test.jmx \
  -Jthreads=50 -Jramp_up=15 -Jduration=180 \
  -l result-50.jtl -e -o report-50/

# 100 并发
jmeter -n -t jmeter/atlas-hot-bank-stress-test.jmx \
  -Jthreads=100 -Jramp_up=30 -Jduration=180 \
  -l result-100.jtl -e -o report-100/

# 200 并发
jmeter -n -t jmeter/atlas-hot-bank-stress-test.jmx \
  -Jthreads=200 -Jramp_up=60 -Jduration=180 \
  -l result-200.jtl -e -o report-200/
```

记录每个阶梯的 Error%、P95，找到「错误率持续 > 1%」的拐点，就是系统在该配置下的极限并发。

## 六、报告解读：哪些指标写进简历

打开 `report-xxx/index.html`，关注以下面板：

### 6.1 Dashboard（首页大盘）

- **APDEX**：满意度评分（1.0 = 全部满意），有缓存组应接近 1.0
- **Requests Summary**：总请求数 / 失败数 / 失败率（Error%）

### 6.2 Statistics（统计表）

| 指标 | JMeter 中对应 | 简历写法 |
|------|-------------|---------|
| Throughput | `Throughput`（req/s） | 「吞吐量 xxx req/s」 |
| 平均 RT | `Average`（ms） | 辅助参考，不作为主要论据 |
| 中位数 RT | `Median`（ms） | 「中位数延迟 xx ms」 |
| P95 延迟 | `90th / 95th pct` | **「P95 延迟从 xx ms 降至 yy ms」**（核心对比） |
| P99 延迟 | `99th pct` | 尾部最差情况 |
| 错误率 | `Error %` | 「错误率维持 0%」 |

### 6.3 Over Time（时序图，面试加分）

- **Response Times Over Time**：横轴时间、纵轴延迟，能看到「冷启动尖刺」逐渐回落的曲线
- **Active Threads Over Time**：并发爬升过程

## 七、常见问题

### Q: JMeter GUI 模式打开 .jmx 后提示缺少 JSON 断言插件？
A: JMeter 5.4+ 已内置 JSON Path Assertion，无需额外插件。如果报错，升级 JMeter 到 5.6+。

### Q: 压测时本机 CPU 飙满，JMeter 本身成为瓶颈？
A: 正常。单机 JMeter 并发能力有限（约 200~500 线程）。如果被测应用 CPU/内存还很宽松但 JMeter 先满，可以：
- 降低线程数到应用能体现差异的区间
- 或换分布式压测（多台机器跑 JMeter Server）

### Q: 实验乙错误率 100%？
A: 检查 MySQL `max_connections` — 100 线程全部穿透到 MySQL 会瞬间占满连接池。先确认 `SHOW VARIABLES LIKE 'max_connections'`，调大到 200+。

### Q: curl 正常但 JMeter 报 500？
A: 看 `View Results Tree` 中具体响应的 `message` 字段。常见原因：
- 数据库连接耗尽 → 调大 HikariCP `maximumPoolSize`
- Redis 断连 → 确认 `redis-cli ping` 通

### Q: 怎么只看「冷启动」那段的表现？
A: 实验乙中删 Key 后立即压测，在 Dashboard 的 Response Times Over Time 图中能看到**前 30~60 秒的延迟尖刺**，之后逐渐平稳（缓存写回）。截图时截这段尖刺 + 后续平稳期做对比说明。
