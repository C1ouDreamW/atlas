# 📑 项目说明文档：基于大模型的智能在线刷题平台

## 1. 项目背景与目标 (Background & Objectives)

**项目起源：** 本项目用于本人的后端开发人员实习简历和面试。

**业务痛点：** 大学期末备考周，学生手中往往有大量散乱的复习资料（Word 题库、PDF 历年真题等）。传统的在线刷题软件需要手动一道道录入题目，耗时极长，且无法满足考前突击的即时需求。

**解决方案：** 开发一款面向本校学生的专属在线刷题网站。系统最核心的壁垒在于引入 LLM（大语言模型）实现多模态文档的"一键智能解析与导入"，并提供高并发场景下的流畅刷题体验。

**核心价值（简历包装点）：** 解决非结构化文档的结构化提取痛点；解决期末高频访问下的数据库读写压力。

## 2. 系统核心模块 (Core Modules)
整个系统主要拆分为四大核心业务模块：

### 2.1 用户鉴权模块 (User & Auth Module)

- **注册/登录：** 支持账号注册登录。
- **安全机制：** 基于 JWT (JSON Web Token) 实现无状态登录校验；密码使用 BCrypt 加密存储。

### 2.2 智能题库导入模块 (AI-Powered Import Module) [⭐ 核心亮点]

- **文件上传：** 支持 .docx, .pdf, .txt 等格式题库文件上传。
- **智能解析：** 后端提取文件纯文本，调用 MinerU API 将文档转换为Markdown文本（异步处理），再调用大模型 API（如 Kimi / DeepSeek），通过精准的 Prompt 设计，将杂乱的文本提取为标准的 JSON 格式（题干、选项、正确答案、解析）。
- **异步处理：** 针对长文档解析较慢的问题，采用异步任务处理，防止前端请求超时。

### 2.3 题库与试题管理模块 (Question Bank Management)

- **题库管理：** 创建、修改、删除题库（如："2026计算机网络期末必刷题"）。
- **试题 CRUD：** 支持手动对解析后的单道题目进行增删改查。

### 2.4 刷题与错题本模块 (Practice & Wrong-Book Module)

- **在线刷题：** 支持顺序刷题、随机抽题。前端页面交互（选中选项、判断对错、展示解析）。
- **错题收集：** 用户做错的题目自动加入"专属错题本"，支持针对错题进行二次重刷。
- **高频缓存：** 针对全校共享的公开题库，引入 Redis 进行缓存，应对期末突击的高并发查询。

## 3. 技术选型与架构 (Tech Stack)

- **后端开发语言：** Java 17
- **核心框架：** Spring Boot 3.x
- **数据持久层：** MyBatis-Plus + MySQL 8.x
- **缓存与性能优化：** Redis (处理高频读请求)
- **文件存储：** 本地磁盘存储
- **AI 大模型服务：** DeepSeek / Kimi / 智谱 GLM (REST API 接入)
- **构建与接口文档：** Maven + Swagger + SpringDoc

## 4. 核心业务流程图解 (Business Workflows)

为了让你开发时思路更清晰，重点理解以下两个核心流程：

### 🌊 流程 A：AI 智能导入流程

- 用户在前端上传一份杂乱的 期末真题.pdf（或其他可接受的格式）。
- Java后端（生产者）接收后，将任务存入Redis Stream中，并返回任务ID给前端。
- Python后端（消费者）从Redis Stream中拉取任务，并调用 MinerU API 将文档转换为Markdown文本（异步处理）。
- Python后端将Markdown文本发送给大模型API，并返回标准的题目 JSON 数组（异步处理）。
- Java后端解析JSON，先返回前端预览，用户确认后，再批量插入 (Batch Insert) 到 MySQL 的试题表中。

### 🌊 流程 B：期末周高并发刷题流程

- 用户点击"开始刷题（高数重点题库）"。
- 后端收到请求，首先去 Redis 中查询该题库的题目数据（Key: question_bank:1001）。
- 如果 Redis 命中，直接返回数据给前端（毫秒级）。
- 如果 Redis 未命中（缓存穿透/失效），则查询 MySQL 数据库，查到数据后将其写入 Redis，并设置过期时间，最后返回给前端。

## 5. 面试加分项/技术难点预留 (Resume Goldmine)

在做这个项目时，我们要刻意把以下几个"故事"做扎实，面试时就能侃侃而谈：

- **复杂非结构化数据处理：** 面试官问你遇到过什么难题，你可以说"传统正则解析 Word 题库容错率太低，我独立构思了结合 LLM 的多模态清洗方案，将导入成功率从 40% 提升到了 95%"。
- **大表查询与联合索引优化：** 错题本表 (wrong_question) 随着时间推移数据量会变大。你通过为 user_id 和 question_id 建立联合索引，优化了查询速度。
- **接口响应时间优化 (Redis)：** 使用 Redis 缓存热点题库，缓解了期末突击时的 MySQL 读压力。
- **异步任务处理：** 使用 Redis Stream 实现异步任务处理，提高系统吞吐量。
- **大模型API集成：** 集成 DeepSeek / Kimi / 智谱 GLM 等大模型API，实现非结构化文本到结构化题库的自动化转换。
